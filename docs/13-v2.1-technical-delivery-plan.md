# 13 — TrainBuffer v2.1 Technical Delivery Plan

**Project:** TrainBuffer
**Repo:** `train-buffer-advisor`
**Status:** v2.0 released (Phases 1–20 done: v1 core, better UI, live weather, live delay check, reliability board, connection mode). Tagged `v1.0` and `v2.0`.
**This document covers:** the next milestone (working name **v2.1**) — production-readiness and the switch from sample data to real Deutsche Bahn statistics (the remaining P2-1 / P2-3 backlog).

> Naming: "v2.0" already shipped all the *features* in `docs/12`. v2.1 is about making the app trustworthy in production (safety net + real data) rather than adding new user-facing modes.

---

## 0. How to use this document

Same developer lifecycle as v1/v2 (`trainbuffer_technical_delivery_plan.md` section 5):

```text
1. Define → 2. Design → 3. Implement → 4. Test → 5. Review → 6. Commit → 7. Next feature
```

Rules (unchanged):

- **One phase at a time**, in order (21, then 22, …). No next feature before the current one passes `pytest`.
- Add tests for the phase, run `pytest`, fix failures, **then** commit.
- Update `docs/progress.md` after every phase in the same commit.
- Never combine phases into one commit; for any phase split into sub-phases, one commit per sub-phase.
- Keep `app.py` thin — orchestration only, business logic in `src/*`.
- **No live API dependency that breaks the app when the API is unavailable** (guardrail carried from v1/v2 — see section 6).

## 1. Phase 21 — Continuous integration + environment pinning

### Goal

Stop broken code (or a broken deploy) from reaching the live app again — directly motivated by the stale-deploy ImportError incident after v2.0.

### Scope

```text
- Add a GitHub Actions workflow that runs on every push and PR to main:
    - sets up a pinned Python version
    - installs requirements.txt
    - runs pytest
    - runs an import smoke check: python -c "import app"
- Pin the Python version used by CI (and document the version the Streamlit
  Cloud app should be set to, so local / CI / deploy all match).
- Add a CI status badge to README.md.
```

### Out of scope

```text
- Deploy automation / CD (Streamlit Cloud already auto-deploys on push)
- Test coverage gates or linting rules (can come later)
```

### New files

```text
.github/workflows/ci.yml
runtime.txt   # or equivalent Python-version marker, if adopted
```

### Done when

```text
- CI runs green on a push (pytest + import smoke check)
- Python version is pinned and documented
- README shows a passing CI badge
- pytest still passes locally
```

### Commit

```bash
git commit -m "Add CI workflow and pin Python version"
```

## 2. Phase 22 — Cache and harden live API calls

### Goal

Now that weather and upstream-delay lookups are real network calls, make them fast and rate-limit-safe without changing behavior.

### Scope

```text
- Wrap fetch_weather_signal / fetch_live_delay results with a short-TTL cache
  (e.g. st.cache_data in app.py, or a thin cache layer in src/*) so repeated
  submits for the same station/train don't re-hit the API.
- Confirm existing timeouts (5s) and fail-closed fallback still hold with cache.
- Add a small, explicit cache TTL constant (documented, testable).
```

### Design notes

```text
- Caching must not defeat the fallback: a failed call must not be cached as a
  permanent failure. Cache successful responses only, or use a very short TTL.
- Keep network isolation intact (_fetch_current_weather / _request_train_status
  remain the only network functions, still mockable).
```

### Tests

```text
- A cached lookup does not call the network function a second time within TTL
  (assert the isolated network function is invoked once for repeated calls).
- A failed call is not served from cache as a success on the next attempt.
- Existing weather / delay tests remain unchanged and passing.
```

### Done when

```text
- Repeated identical lookups hit the network at most once per TTL
- Fallback behavior unchanged and still tested
- pytest passes
```

### Commit

```bash
git commit -m "Cache and harden live weather and delay lookups"
```

## 3. Phase 23 — Real Deutsche Bahn historical data (P2-1)

### Goal

Replace the 5-row hand-curated `data/sample_station_stats.csv` with real aggregated station-reliability statistics. This is the single biggest accuracy/credibility gain and unblocks the whole app (advisor, reliability board, connection mode all read `StationStats`).

### Sub-phases (do not combine into one task)

```text
23a. Data-source decision + acquisition
     - Pick a source (e.g. DB open-data / DB Timetables aggregation, or a
       published punctuality dataset). Document terms of use and refresh method.
     - Store a raw snapshot under data/ (do not fetch live at runtime).

23b. Ingestion / aggregation script
     - New file: src/data_ingest.py (or scripts/)
     - Pure, tested function that maps raw source rows to StationStats fields
       (station_name, sample_size, late_rate, cancellation_rate,
       avg_delay_minutes, p80_delay_minutes). Validate ranges (reuse model
       validation). Output a real data/station_stats.csv.

23c. Swap the data source in the app
     - Point STATION_STATS_PATH at the real dataset.
     - Keep the sample CSV as a documented fallback if the real file is missing.
     - Update the data-freshness indicator to reflect the real dataset date.
```

### Tests

```text
- ingestion maps a known raw row to the correct StationStats values
- ingestion rejects / flags malformed rows (out-of-range rates, negative sizes)
- an empty or missing source degrades gracefully (documented behavior)
- load_station_stats still returns valid StationStats for the real file
- existing engine tests remain green against the model (unchanged logic)
```

### Design notes / guardrails

```text
- The risk engine, recommendation text, board, and connection engine MUST NOT
  change — only the data behind StationStats changes (same as the weather-source
  swap in Phase 17: change the source, not the rule).
- No runtime dependency on a live data feed: ingestion is an offline/batch step
  producing a CSV the app reads. The app must still run if the feed is down.
- If real data can't be obtained, document the blocker and keep the sample data;
  do not ship fabricated numbers.
```

### Done when

```text
- app reads a real (or realistically aggregated) station dataset
- ingestion is a tested pure function, not ad-hoc one-off code
- data source + refresh cadence documented in README and docs/progress.md
- pytest passes
```

### Commit

```bash
git commit -m "Add real DB historical data: <sub-phase name>"
```
(one commit per sub-phase 23a, 23b, 23c)

## 4. Phase 24 — Real construction/disruption data (P2-3) — spike, may defer

### Goal

Revisit the construction/disruption signal that was deliberately kept manual in Phase 19. Decide, with a time-boxed spike, whether a stable low-complexity source now justifies replacing the manual flag.

### Scope

```text
- Time-boxed investigation only (no half-built integration merged).
- If a viable source is found: implement it behind the same guardrails as
  Phase 17/18 (isolated, mockable, fail-closed), reusing apply_construction_
  modifier unchanged.
- If not: update the documented deferral in docs/progress.md with findings and
  stop. This is an acceptable outcome.
```

### Done when

```text
- A written decision (adopt vs. keep manual) with reasons is committed
- If adopted: implemented with mocked tests and fail-closed fallback; pytest passes
```

### Commit

```bash
git commit -m "Decide construction/disruption data source (adopt or document deferral)"
```

## 6. Guardrails (carried over — still apply)

```text
Rule 1 — No new feature before the app runs locally on the current changes
Rule 2 — No UI before the underlying logic has tests
Rule 3 — No ML before the rule-based baseline is proven insufficient
Rule 4 — No live API dependency that breaks the app when the API is unavailable
Rule 5 — Every feature must improve the core decision: "what is my latest safe
         planned arrival, and should I take one earlier?"
```

---

## 7. Suggested order and rationale

```text
21 (CI + pin)      first  — cheap safety net; protects everything after it
22 (cache/harden)  next   — small, removes the main risk of the new live calls
23 (real data)     core   — the actual product upgrade; largest effort
24 (construction)  spike  — bounded; may end in a documented deferral
25 (release)       last   — only meaningful once real data is in
```

## 8. Definition of Done for this document's scope

```text
- CI runs pytest + import smoke check on every push (Phase 21)
- Live API calls are cached and still fail closed (Phase 22)
- The app runs on real aggregated station data with a tested ingestion step,
  and the engine logic is unchanged (Phase 23)
- Construction/disruption data decision is documented (Phase 24)
- v2.1 release is tagged with notes and README reflects the live UI (Phase 25)
- docs/progress.md updated after every phase, same as v1/v2
```
